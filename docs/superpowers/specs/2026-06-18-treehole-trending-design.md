# PKU Treehole 热帖追踪 — 产品设计规格

> 2026-06-18 | v1.0

## 1. 概述

从北大树洞 (treehole.pku.edu.cn) 实时追踪热帖，基于收藏数、评论数、独立评论人数三项指标，按时间窗口输出热度排名。

### 时间窗口

| 窗口 | 数据量 | API 耗时 |
|------|--------|---------|
| 1 小时 | ~100 帖 | <1 秒 |
| 半天 | ~1200 帖 | ~12 秒 |
| 1 天 | ~2400 帖 | ~24 秒 |
| 3 天 | ~7200 帖 | ~72 秒 |
| 1 周 | ~16800 帖 | ~3 分钟 |

默认 Top 10，窗口可通过参数切换。

## 3. 热度算法

### 第一阶段：粗筛

遍历 list API，按简单加权分取 Top 100：

```
coarse_score = likenum × 2 + reply × 3
```

- 数据来源：`/chapi/api/v3/hole/list`（无需额外请求）
- 遍历范围：按时间窗口回溯，逐页拉取直到 `timestamp` 超过窗口边界
- 请求参数：`limit=100`（单页最大），`page=1, 2, 3...`

### 第二阶段：精排

对 Top 100 逐帖拉取评论 API，获取独立评论人数 U：

```
base_score = U × 5  +  ln(reply + 1) × 3  +  ln(likenum + 1) × 2

bonus(t) = 5.0 × max(0, 1 - t)        # t = 距现在的小时数

final_score = base_score × (1 + bonus(t))
```

| 参数 | 值 | 说明 |
|------|-----|------|
| W_u | 5 | 独立评论人数权重（最重要） |
| W_c | 3 | 评论总数权重（ln 缩放，次重要） |
| W_l | 2 | 收藏数权重（ln 缩放） |
| B | 1.0 | 最大时间加成倍数 |
| t_cutoff | 1 小时 | 加成临界值 |

**时间加成特性：**

- `t = 0`（刚刚发帖）：bonus = 1.0，分数翻倍
- `t = 0.5`（半小时）：bonus = 0.5
- `t = 1`（1 小时）：bonus = 0，连续衔接
- `t > 1`：bonus = 0，无衰减，老帖不贬值

### 数据来源映射

| 算法符号 | 字段名 | 来源 API |
|----------|--------|---------|
| likenum | `likenum` | list API（均有） |
| reply | `reply` | list API（均有） |
| U | count(unique `name`) | `/api/pku_comment_v3/{pid}` |

### 评分示例

假设帖子（发帖 0.3 小时前）：

```
likenum = 10, reply = 25, U = 8

base_score = 8×5 + ln(25+1)×3 + ln(10+1)×2
           = 40 + 9.78 + 4.80 = 54.58

bonus(0.3) = 1.0 × (1 - 0.3) = 0.7

final_score = 54.58 × 1.7 = 92.79
```

## 4. API 设计

### 后端接口

```
GET /api/trending?window=1d&limit=10
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| window | string | "1d" | 时间窗口：1h, 0.5d, 1d, 3d, 7d |
| limit | int | 10 | 返回数量 |

**响应：**

```json
{
  "window": "1d",
  "generated_at": 1781717000,
  "posts": [
    {
      "pid": 8323082,
      "text": "帖子正文...",
      "timestamp": 1781713646,
      "likenum": 4,
      "reply": 42,
      "unique_commenters": 4,
      "final_score": 92.79,
      "rank": 1
    }
  ]
}
```

### 健康检查

```
GET /api/health
→ { "status": "ok", "treehole_connected": true }
```

## 6. 数据采集流程

```
用户请求 /api/trending?window=7d&limit=10
         │
         ▼
    ┌─────────────┐
    │ 确定时间边界  │  now - window
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 遍历 list API│  page=1,2,3... → 累积 posts
    │ 直至超时边界  │  每页 limit=100, delay=1s
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 第一阶段粗筛  │  coarse_score = likenum×3 + reply×1
    │ Top 100     │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 第二阶段精排  │  对 Top 100 逐帖:
    │             │  1. GET /api/pku_comment_v3/{pid}
    │             │  2. U = count(unique name)
    │             │  3. 计算 final_score
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 排序返回     │  Top N, 按 final_score 降序
    └─────────────┘
```

### 请求量预估

- 第一阶段 list API：窗口越大请求越多，最多 ~170 次（1 周）
- 第二阶段 comment API：固定 100 次（Top 100）
- 总计：最长 270 次请求，~4.5 分钟

## 7. 缓存策略

为改善较长窗口的等待体验，引入内存缓存：

```
- list API 结果缓存 30 秒
- 单帖评论 U 值缓存 2 分钟
- 同一窗口热门结果缓存 30 秒
```

用户刷新时，1 小时窗口基本瞬间返回，1 周窗口最差 4.5 分钟。
